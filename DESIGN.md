# compriere 設計書

くじを回した回数ごとのコンプリート確率を計算する Web アプリの設計（現在の状態）。
時系列の経緯・作業記録は [DEVLOG.md](DEVLOG.md) を参照。

## 目的とコア価値

- **確率計算エンジン**: 「1賞に複数のランダム景品があるくじ」のコンプ率を、回数ごとに実用的な速度で計算する。本アプリの核。
- **実測ベースの提供割合**: 非公開のウェブポンの提供割合を実測（実際に引かれた個数 約20万件）から推定したプリセットを同梱。
- **サーバーレス**: GitHub Pages で完結（バックエンドなし）。

プロダクト方向は「汎用コンプ率計算機」。誰でも賞・種類数・確率を編集できるが、実測プリセットを目立つ位置に置き、ウェブポン目的の流入も維持する。

## アーキテクチャ

3層に分離し、計算ロジックを UI から独立させている（テスト可能・型安全）。

```
src/
  core/    DOM非依存の純粋ロジック（型・テストあり）
    completion.ts   コンプ率／1個以上出る率の計算エンジン
    legacy-dp.ts    旧サイトのDP（検証用の参照実装。本番不使用）
    lottery.ts      Lottery / Tier 型 ＋ parseLottery（実行時バリデーション）
    evaluate.ts     Lottery × 回数 → 賞ごとの結果
    format.ts       パーセント表示
  data/
    index.ts        プリセットの読み込み・検証
    presets/level1〜5.json   実測くじ（人気度別の5段階）
  ui/
    LotteryTable.tsx  編集テーブル＋結果表示
  App.tsx / main.tsx  状態管理と画面構成
```

Vite プロジェクト＝リポジトリ直下（単一アプリ構成）。スタックは React + TypeScript（strict）+ Vite、テストは Vitest。

## 計算エンジン

モデル: 1回引くごとに確率 p で当たり、当たりは k 種類から一様（特定の1種類は p/k）。これを n 回引く。**賞ごとに独立**に計算する。

- **コンプ率**（n回で k 種類すべて集まる確率）: k の大小で2手法を使い分けるハイブリッド。
  - `k ≤ 20`: 包除原理 `Σ_{j=0}^{k} (-1)^j C(k,j) (1 - j·p/k)^n`。O(k)、n がいくつでも一瞬。
  - `k > 20`: 占有数DP（`dp[j] = ちょうど j 種類集めた確率` を1回ずつ更新）。O(n·k)、二項係数を使わず全状態が [0,1] に収まり桁落ちしない。
  - 使い分けの理由: 包除原理だけだと k が大きいとき各項が C(k, k/2) まで巨大化し（k=92 で約 3.6×10²⁶）、符号交替の打ち消しで桁落ちして答えが壊れる。詳細は DEVLOG 2026-06-28。
- **1個以上出る率**（賞全体）: `1 - (1-p)^n`。

正しさは `legacy-dp.ts`（旧サイトのDP）を参照実装として、広範な (n,k,p) で数値一致をテストして担保している。

## データモデル

```ts
type Tier = { label: string; rate: number; kinds: number }; // rate は 0..1
type Lottery = {
  id: string;
  name: string;
  description?: string;   // 出典・補足（任意）
  tiers: Tier[];          // 可変長
};
```

- `parseLottery(unknown)` で実行時バリデーション（プリセットJSON・ユーザー編集など信頼できない入力に備える）。エラーは全件まとめて返し、UI で表示できる。
- 提供割合の合計は本来100%以下（1回の抽選で出る賞は排他）。実測・手入力のパーセント丸めを考慮し `RATE_SUM_TOLERANCE = 0.01`（約1%）まで許容し、超過のみ警告・拒否する。
- コンプ率は賞ごとに独立。「全賞まとめて全アイテムを集める」コンプは別物（計算が重い）で対象外。

## プリセット

`data/presets/level1〜5.json`。引かれる総数（人気度）が多いガチャほど上位賞が渋くなる関係を、人気度別の5段階で表現。初期表示は level3「普通」。`data/index.ts` がロード時に `parseLottery` で検証して公開する。実測値の更新は該当 JSON を編集するだけ。

## デプロイ

- GitHub Actions（`.github/workflows/deploy.yml`）が main への push（または手動実行）で `npm ci → npm test → npm run build` し、`dist/` を Pages へ配信。テストが落ちるとデプロイされない。
- 本番ビルドのみ `base: '/compriere/'`（プロジェクトページ配下）。dev は `/`。
- GitHub Pages の Source は「GitHub Actions」。（「Deploy from a branch」だと組み込みのルート配信と競合して旧サイトに戻るため不可。）

## 主要な決定と理由

- **包除原理 → ハイブリッド**: 旧DPは二項係数の Number 桁あふれで n>1000 が限界。包除原理で n に強くしたが k で桁落ちするため、k>20 は占有数DPに切替えて両対応。
- **完全ユーザー設定型 ＋ 実測プリセット**: 汎用性と既存の検索流入の両取り。
- **データをコードから分離（JSON）**: くじ追加・実測更新をデータ変更だけで完結させ、保守性を上げる。
- **strict TS ＋ 実行時バリデーション**: 旧実装の「型なし」を解消し、ユーザー入力に堅牢化。
- **URL共有／localStorage は当面スコープ外**: 利用見込みが低い。UI状態は App に集約済みで後付けは容易。

## スコープ外・将来候補

- URL共有・localStorage によるくじの保存／共有。
- 賞をまたいだ「全アイテム」コンプ率。
- 賞内で種類ごとにレートが異なる per-item モデル（現状は賞内一様）。
- UI のレンダリングテスト（jsdom + testing-library）。
